# Context Glossary

## Terms

- **Agent workspace**: the OpenClaw working tree that holds agent memory, hooks, persona files, experiments, manual helpers, and other workspace-specific runtime concerns for one agent. Does not hold integration secrets; **host credentials** live outside the workspace.
- **Host credentials**: API keys, tokens, and similar secrets for production integrations, supplied via the droplet environment and the infrastructure repository (`UbiClawBot`), not stored in agent-owned files.
- **Agent workspace vault**: one Obsidian vault (one remote Sync vault) used as the human-facing place to read and edit agent-owned files across all agents. Not versioned through git; kept current by automatic two-way sync between runtime and Obsidian (headless on the server, desktop app on personal devices). The droplet runtime copy is authoritative when migrating; local git clones of agent repos are not. After cutover, superseded runtime directories are renamed (not deleted immediately) until smoke tests pass. Sync collisions produce conflict copies rather than silent overwrites. Vault root layout: a human hub note; **`wiki/`**; and one subdirectory per **agent workspace** (`ubi/`, `cheryl/`, `marcos/`), named for the agent in lowercase, not for OpenClaw agent IDs. In-repo path references use `agent-vault/{agent}/…` (and `agent-vault/wiki/…`), not legacy `workspace*` directory names.
- **raw input**: a flat vault inbox at `raw-input/` (sibling to `wiki/` and agent workspaces)—the only place Ubi, Cheryl, and Marcos may drop new or revised **produced artifacts** at runtime. Files live directly under `raw-input/` (no per-agent subfolders in v1). v1 filename convention: `{agent}-{YYYY-MM-DD}-{slug}.md` (agent slug and date required; slug describes content). The **wiki curator** chooses the final path under `wiki/` from content and filing rules—not from a bucket encoded in the filename. To change wiki content, producers add a file here (including an edited copy of an existing wiki page); they do not edit `wiki/` in place. **Wiki curation** moves each processed drop to its target path under `wiki/` (or merges into an existing **vault document**, then removes the drop). Failed or ambiguous drops stay in `raw-input/` or move to `raw-input/_failed/`. **Raw input** is synced to the operator’s Obsidian devices for visibility but excluded from the operator RAG corpus (same idea as `openclaw-docs/` exclusion) until filed into **wiki**.
- **wiki curator**: Cheryl (OpenClaw agent `scheduler`), trained to process **raw input** and organize filed content into **wiki/**. Cheryl is the only agent that may create, move, merge, or delete **vault documents** under `wiki/` at runtime (except `openclaw-docs/`). In v1 the curator runs on **cron** only, on a short fixed cadence (approximately every fifteen minutes), using the **`opencode-go/deepseek-v4-flash`** model (see `docs/deployment/openclaw-agents.md`). The operator edits **wiki/** directly in Obsidian; operator edits do not go through **raw input**. All agent **producers** (including Cheryl’s scheduling work) drop **produced artifacts** into **raw input** only; curator turns file them into **wiki/**.
- **wiki**: the **Obsidian document corpus** inside the **agent workspace vault** (`wiki/`)—curated cross-agent markdown for humans and agents to read. Not a substitute for **host credentials**, pipeline config, or per-agent persona/memory. At runtime, **vault documents** live here after Cheryl files them from **raw input**; producer agents read `wiki/` but do not write it. `openclaw-docs/` remains a droplet-only mirror excluded from Obsidian Sync and the operator RAG corpus. Sync uses conflict copies, not silent overwrites.
- **runbook**: a stable **vault document** under `wiki/runbooks/`—practices, checklists, how-to, and agent-specific rulebooks (e.g. Marcos maintenance law). Distinct from **workflow** documents under `wiki/workflows/` (living integration maps: cron summaries, Trello/calendar wiring, helper registry, runtime layout).
- **Marcos rulebook**: Marcos’s durable maintenance instructions as three **vault documents** under `wiki/runbooks/marcos/`: `master.md`, `scope-rules.md`, `nightly-prompt.md`. The `marcos/` **agent workspace** holds only OpenClaw machinery (persona, `memory/`, `skills/`, a short `AGENTS.md` that points at the rulebook and report paths)—not duplicate copies of the rulebook or dated audit outputs.
- **vault document**: any **produced** artifact (reports, runbooks, job materials, project writeups, screenshots with context, research, admin notes, etc.) that is not required as OpenClaw workspace machinery for a specific agent. Before curation it appears under **raw input**; after curation it lives under **wiki/** for the Obsidian document corpus. Distinct from per-agent **agent workspace** files OpenClaw needs in that agent’s tree: persona roots (`AGENTS.md`, `SOUL.md`, …), `memory/`, `skills/`, `hooks/`, `taskflow/`, agent-specific CSVs, and similar.
- **produced artifact**: output or durable reference material created by an agent or operator during work (nightly report, application draft, audit, export note)—as opposed to configuration or state OpenClaw loads by convention from the agent workspace.
- **wiki top-level folder**: one of the synced document buckets under `wiki/` (lowercase, no spaces): `reports/`, `runbooks/`, `job-search/`, `personal/`, `projects/`, plus `workflows/` for cross-agent operational writeups (Trello/calendar, cron summaries, helper registry, runtime layout). No separate `exports/` bucket—screenshots and captures are **embedded** in the markdown **vault document** they support (report, job-search note, project page). `openclaw-docs/` is a separate top-level tree—agent mirror only, excluded from Obsidian Sync and the operator RAG corpus.
- **dated report folder**: under `wiki/reports/`, a subdirectory named `YYYY-MM-DD` holding agent-generated dailies (e.g. nightly job search). One-off research and audits live at `wiki/reports/` root or in topic subfolders (e.g. `repo-maintenance/`), not under date folders unless tied to a specific day.
- **legacy export capture**: screenshot/PNG files from the old `Docs/Exports and Backups/` tree are not migrated into the vault; they remain only in the `workspace.old` backup (or are discarded). New captures are embedded in markdown **vault documents**, not stored as standalone export files.
- **job-search document**: a **vault document** for applying—résumés, cover letters, application drafts, follow-ups, application status. Lives under `wiki/job-search/`. Distinct from **job-search report** (nightly analysis, shortlists, market research, job-related RCAs under `wiki/reports/` or `wiki/reports/YYYY-MM-DD/`).
- **wiki bootstrap**: one-time import of legacy `Docs/` and stray `wiki/` root files into the agreed `wiki/` layout by deterministic copy (not via **raw input**), so the **wiki curator** is not overloaded on day one. After bootstrap, new **produced artifacts** from agents use **raw input** only.
- **wiki migration dedup**: during **wiki bootstrap** (and when the curator files a drop), each file lands in its target **wiki top-level folder**; if a copy already exists elsewhere in `wiki/`, diff/merge content, then remove the duplicate—do not preserve pre-layout paths for convenience.
- **Trello production pipeline**: the production code path that receives Trello events, classifies them, queues unattended work, runs Trello/calendar automation, and applies resulting Trello/calendar changes.
- **Scheduled Trello/calendar job**: an unattended Trello/calendar automation job triggered by time or cron rather than by an incoming webhook event.
- **Pipeline state**: durable operational state used by the Trello production pipeline, such as queue records, handled-action tracking, retry tracking, and similar service state. This is distinct from agent memory or taskflow state.
- **Documentation contract test**: a `node:test` case in `UbiClawBot` that asserts a structural or cross-repo documentation invariant (for example, a forbidden duplicate path is absent or a required canonical page exists). These tests may read sibling agent repositories; they are not tests of runtime Trello behavior.

## Trello board language

- **open-card contract**: the gateway-enforced shape of cards on scoped lists—four description sections in fixed order (`Original Request`, `Research`, `Peer Review`, `Work completed`) plus section mutation rules (e.g. Marcos-only `Peer Review`). Distinct from agent prompts and skills.
- **backlog intake**: the procedure Ubi runs when a new or moved card enters **Backlog**—style the card, fill the open-card contract sections, then execute, block, or hand off. Not applied to **PR review cards**.
- **peer review**: optional feedback from Marcos on a backlog card Ubi judges technical (code, infra, deploy, security, architecture). Ubi @mentions Marcos and leaves the card in **Backlog** until he replies; not a **Blocked** state.
- **PR review card**: a Trello card created by `github-pr-bridge` for an open pull request (title pattern like `Review PR #N`). Lands in **Backlog**; owned by Marcos for the GitHub review; Ubi backlog intake is skipped.
- **Next steps checklist** *(retired)*: a native Trello checklist formerly required by the open-card contract. No longer enforced; open cards may still carry legacy copies pending a one-time cleanup sweep (skip **Done** and Trello-closed cards).
