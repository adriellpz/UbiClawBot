# Context Glossary

## Terms

- **Agent workspace**: the OpenClaw working tree that holds agent memory, hooks, persona files, experiments, manual helpers, and other workspace-specific runtime concerns for one agent. Does not hold integration secrets; **host credentials** live outside the workspace.
- **Host credentials**: API keys, tokens, and similar secrets for production integrations, supplied via the droplet environment and the infrastructure repository (`UbiClawBot`), not stored in agent-owned files.
- **Agent workspace vault**: one Obsidian vault (one remote Sync vault) used as the human-facing place to read and edit agent-owned files across all agents. Not versioned through git; kept current by automatic two-way sync between runtime and Obsidian (headless on the server, desktop app on personal devices). The droplet runtime copy is authoritative when migrating; local git clones of agent repos are not. After cutover, superseded runtime directories are renamed (not deleted immediately) until smoke tests pass. Sync collisions produce conflict copies rather than silent overwrites. Vault root layout: a human hub note; **`wiki/`** (including **`wiki/sources/`** for immutable **raw sources**); **`raw-input/`** (agent publishing queue); and one subdirectory per **agent workspace** (`ubi/`, `cheryl/`, `marcos/`), named for the agent in lowercase, not for OpenClaw agent IDs. In-repo path references use `agent-vault/{agent}/…` (and `agent-vault/wiki/…`, `agent-vault/raw-input/…`), not legacy `workspace*` directory names.
- **raw sources**: the immutable source layer at `wiki/sources/` — articles, papers, web clips, and other reference material the operator (or clipper tools) adds. **`source assets`** are created automatically during **wiki ingest** (not by the operator). Cheryl reads **raw sources** during ingest (text first, then local **source assets**); she never edits source markdown. Distinct from **raw input**. Ingested markdown sources tracked in **`wiki/sources/ingested.log`**. Pending ingest = any `wiki/sources/**/*.md` absent from `ingested.log` (excluding `ingested.log` and `wiki/sources/assets/**`).
- **source assets**: local image files under `wiki/sources/assets/{source-slug}/`, **auto-downloaded on wiki ingest** from URLs in the matching **raw source** clip (script Cheryl runs before or as part of source ingest). Operator clips to `wiki/sources/` only — no manual attachment step. Cheryl reads assets when integrating into **wiki**; important images are **embedded** in **vault documents** (not hotlinked). Original source markdown stays unchanged; asset files are a ingest-time sidecar. Excluded from **wiki log** and operator RAG until synthesized.
- **raw input**: a flat vault inbox at `raw-input/` (sibling to `wiki/` and agent workspaces)—the only place Ubi, Cheryl, and Marcos may drop new or revised **produced artifacts** at runtime. Files live directly under `raw-input/` (no per-agent subfolders in v1). v1 filename convention: `{agent}-{YYYY-MM-DD}-{slug}.md` (agent slug and date required; slug describes content). To publish a **new** **vault document**, producers drop content without a target; Cheryl chooses the final path under `wiki/` from body and filing rules. To **revise** an existing page, producers drop a **wiki update drop** with `update: wiki/...` frontmatter (see below)—they do not edit `wiki/` in place. Failed or ambiguous drops stay in `raw-input/` or move to `raw-input/_failed/`. **Raw input** is synced to the operator's Obsidian devices for visibility but excluded from the operator RAG corpus until integrated into **wiki**.
- **wiki update drop**: a **raw input** file that revises an existing **vault document**. Must include YAML frontmatter `update: wiki/path/to/page.md` (vault-relative path). The body carries proposed changes or a full replacement; filename stays `{agent}-{YYYY-MM-DD}-{slug}.md`. Cheryl loads the target page, merges intelligently (preserving operator edits where possible), cross-references the **wiki** for contradictions, applies the **page format contract**, may touch related pages, then removes the inbox copy.
- **wiki curator**: Cheryl (OpenClaw agent `scheduler`), trained as the **wiki maintainer** — she reads **raw sources** and **raw input**, integrates them into the persistent **wiki**, and keeps the corpus current over time. On **wiki ingest**, a single source may create or update many **vault documents** (summaries, entity pages, cross-references, contradiction notes). On each cron tick she also runs **wiki maintenance** on a small backlog of pages not yet brought up to standard. Cheryl is the only agent that may create, move, merge, or delete **vault documents** under `wiki/` at runtime (except `openclaw-docs/`). In v1 she runs on **cron** only, six times per day (not every fifteen minutes), using the **`opencode-go/deepseek-v4-flash`** model (see `docs/deployment/openclaw-agents.md`). The operator edits **wiki/** directly in Obsidian; operator edits do not go through **raw input**. All agent **producers** drop **produced artifacts** into **raw input** only; the operator adds **raw sources** to `wiki/sources/`; the curator integrates both into **wiki/**.
- **wiki ingest**: Cheryl's process of reading a pending **raw source** or **raw input** drop, extracting key information, and integrating it into the **wiki** — creating or updating **vault documents**, cross-references, and summaries as needed. A **wiki update drop** (`update:` frontmatter) triggers merge-and-integrate on one target page; a new drop or **raw source** may touch many pages. After successful ingest of a **raw input** drop, the inbox copy is removed; **raw sources** remain unchanged in `wiki/sources/`.
- **wiki maintenance**: Cheryl's ongoing work to keep the **wiki** healthy — applying **page format contract**, adding cross-links, and light **wiki lint** on each page touched. Runs every **curator cron tick** after ingest, up to five pages per run, chosen by **maintenance bucket priority**: `runbooks/` → `workflows/` → `job-search/` → `personal/` → `projects/` → `reports/` (newest dated subfolders first within `reports/`), then any **wiki top-level folder** Cheryl created. Targets **vault documents** not yet listed in the **wiki log** completion registry.
- **wiki lint**: corpus health check — orphan pages (no inbound links), stale claims vs newer sources, missing cross-refs, concepts mentioned but lacking a page, and **contradiction search** (cross-page conflicts). When Cheryl finds a contradiction during lint, she **searches the wiki and sources**, opens a **contradiction record** in **`contradictions/`** with inline callout plus her findings, and adds callouts on affected pages. **Light lint** runs on pages touched during ingest and maintenance. **Full lint** runs on every **fourth idle curator tick** instead of `NO_REPLY`.
- **curator cron tick**: Cheryl's fixed pipeline each ~15-minute run: (1) pre-flight **wiki log** invalidation via mtime, (2) ingest all **raw input** drops, (3) ingest one pending **raw source** (auto-download **source assets**, then integrate), (4) **wiki maintenance** on up to five unlogged pages (includes light lint), (5) regenerate **folder index**, **wiki master index**, and **wiki search** index for touched paths, (6) if work queues and registry are clear → **idle curator tick** (increment **curator idle streak** in agent config; every **fourth** streak run **full wiki lint**; otherwise `NO_REPLY`), (7) append **wiki log** chronicle when any work ran.
- **wiki search**: local hybrid search over the curated **wiki** corpus via **qmd** (Query Markup Documents — on-device BM25 + vector + LLM re-ranking), indexed as the qmd collection named `wiki`. Cheryl and agents use qmd CLI (`qmd search`, `qmd query`) during **wiki ingest**, **wiki lint**, **contradiction search**, and **wiki query** — supplementing **wiki master index** skimming. Index scope: in-scope **vault documents** under `wiki/` (same exclusions as **wiki log**, which excludes **`wiki/openclaw-docs/`**). **`wiki/sources/`** and **`raw-input/`** excluded. **`wiki/openclaw-docs/`** is searchable separately via **openclaw-docs search**, never through wiki search. Ubi may use qmd CLI or qmd MCP from **`AGENTS.md`**.
- **openclaw-docs search**: qmd search over the **`wiki/openclaw-docs/`** platform-documentation mirror (OpenClaw's own product docs — cli, gateway, providers, tools, concepts; ~442 pages), indexed as a **separate qmd collection** named `openclaw-docs`, distinct from **wiki search**. Agent-facing reference lookups only (`qmd query -c openclaw-docs`); deliberately excluded from **wiki search**, **wiki lint**, **contradiction search**, and the operator RAG corpus so platform docs never pollute the curated personal **wiki** or generate cross-corpus contradictions. Exists because openclaw-docs is genuine reference agents may need, but is not part of the curated personal **wiki**.
- **idle curator tick**: a **curator cron tick** with no ingest or maintenance backlog (inbox empty, sources ingested, registry matches wiki set). **Curator idle streak** lives in **Cheryl (`scheduler`) agent config**, not the vault. Every fourth idle tick → **full wiki lint**; any work resets streak to zero.
- **curator idle streak**: consecutive **idle curator ticks** counter in **Cheryl (`scheduler`) agent config**. Every fourth tick triggers **full wiki lint** (including **contradiction search**).
- **wiki publishing schema**: producer-facing workflow doc at `wiki/workflows/wiki-publishing.md` — **interaction capture**, **raw input** drops, **`update:` frontmatter**, **raw sources** (clip only; images auto-download on ingest), and **wiki query** results.
- **interaction capture**: when an agent has a conversation with the operator worth preserving (decision, research, concept, note, or discussion summary), the agent drops a short **produced artifact** into **raw input** before the session ends — `{agent}-{YYYY-MM-DD}-{slug}.md`, concise body with durable facts. Required section in each producer **`AGENTS.md`** (Ubi, Cheryl, Marcos). Cheryl integrates via **wiki ingest** on a later tick — not the same run that wrote the drop.
- **wiki query**: an interactive question-and-answer pass against the **wiki** (operator or Ubi in chat — not a cron job). Read **`wiki/index.md`** first, drill into relevant pages, synthesize an answer with citations. Lasting results use **interaction capture** or a **wiki update drop**. Cheryl integrates on the next **curator cron tick**.
- **contradictions bucket**: top-level **`wiki/contradictions/`** — temporary home for open **contradiction records** until resolved. Cheryl may create this **wiki top-level folder**; files live flat (`wiki/contradictions/{slug}.md`) per **wiki layout depth**.
- **contradiction record**: a **vault document** in **`contradictions/`** opened by Cheryl during **wiki lint** when **contradiction search** finds conflicting claims. Body: inline callout (`> **Contradiction:** [[page-a]] says X; [[page-b]] says Y`) plus Cheryl's search findings and recommended resolution. Removed after **contradiction review**. Same callout on affected pages until resolved.
- **contradiction review**: once-daily **scheduled job** for Cheryl (`scheduler`) — Telegram listing open **contradiction records** (already researched during lint). Operator and Cheryl decide resolution; Cheryl updates **vault documents**, removes callouts, deletes the record, logs in **wiki log** chronicle. Distinct from the ~15-minute **curator cron tick**.
- **wiki curator schema**: Cheryl-facing workflow doc at `wiki/workflows/wiki-curator.md` — **curator cron tick** pipeline, **wiki ingest**, **wiki maintenance**, **wiki lint**, **wiki search** (qmd), **page format contract**, **wiki log**, **`wiki/sources/ingested.log`**, **folder index** and **wiki master index** regeneration, **contradiction review**, and `NO_REPLY` rules. The `cheryl-vault-inbox` skill points here; it is a thin runtime wrapper.
- **wiki log**: the combined curator log at `wiki/log.md` with two sections. **Completion registry** (top): one line per **vault document** at curator standard — `wiki/path/to/page.md<TAB>curator-touch-iso8601`. Pre-flight **removes** entries whose file **mtime is newer than the curator touch** (operator edits invalidate; page returns to maintenance backlog). Used for `NO_REPLY` and backlog detection. **Chronicle** (append-only, below): parseable event lines `## [iso8601] ingest \| source` or `## [iso8601] maintenance \| N pages`, each followed by `- touched:` wiki paths — human timeline; grep-friendly. Cheryl refreshes registry timestamps after she writes; appends chronicle entries each **curator cron tick** that did work. Excluded from the registry contract: `wiki/openclaw-docs/**`, `wiki/sources/**`, `**/*-index.md`, `wiki/log.md`, and `wiki/index.md`. **`NO_REPLY`** on idle ticks 1–3 of each idle cycle; full **wiki lint** on idle tick 4 (then streak resets). Immediate **`NO_REPLY`** only if lint also finds nothing to fix — rare. **Wiki log bootstrap**: registry starts empty at cutover (~152 in-scope pages on strict backlog).
- **folder index**: an Obsidian catalog page `{folder-name}-index.md` inside a wiki directory — wikilinks to sibling `*.md` files with a one-line summary pulled from each page's **page format contract** blurb (format: `- [[slug]] — summary`). Excludes `*-index.md` and `log.md`. Regenerated by the **vault index generator** for **touched folders** at the end of each **curator cron tick**.
- **wiki master index**: the content-oriented catalog at `wiki/index.md` — hierarchical drill-down for LLMs and humans. Top level: **wiki top-level folder** sections; nested `###` subfolder headings mirror the directory tree (e.g. `reports/` → `2026-05-28/` → `repo-maintenance/`). Each page entry: `- [[path/to/page]] — one-line summary`. Folder rows link to the local **folder index** (e.g. `- [[reports/2026-05-28/2026-05-28-index]] — 3 pages`). Regenerated by the same generator pass as **folder index** files (full master rollup each run, or touched-subtree + master). Cheryl and agents read this first when navigating the corpus.
- **page format contract**: the human-readable shape Cheryl enforces on every **vault document** she touches — **YAML frontmatter** block first (`tags`, `created`, `updated`, `type`, optional `sources` list), then `#` one-line title, 3–4 sentence blurb, body, and `## Related` with 0–10 wikilinks. Frontmatter is **in addition to** **wiki master index** and **folder index** files (Dataview can query tags and dates; indexes stay the LLM navigation layer). **`type`** values include report, runbook, workflow, job-search, personal, project, concept, entity, synthesis, contradiction. Cheryl **normalizes always** (refreshes `updated` and envelope fields; sets `created` once; preserves body on operator edits). Exempt: `wiki/openclaw-docs/**`, `wiki/sources/**`, `**/*-index.md`, `wiki/log.md`, `wiki/index.md`.
- **wiki**: the **Obsidian document corpus** inside the **agent workspace vault** (`wiki/`)—curated cross-agent markdown for humans and agents to read. Not a substitute for **host credentials**, pipeline config, or per-agent persona/memory. At runtime, **vault documents** live here after Cheryl files them from **raw input**; producer agents read `wiki/` but do not write it. `openclaw-docs/` remains a droplet-only mirror excluded from Obsidian Sync and the operator RAG corpus. Sync uses conflict copies, not silent overwrites.
- **runbook**: a stable **vault document** under `wiki/runbooks/`—practices, checklists, how-to, and agent-specific rulebooks (e.g. Marcos maintenance law). Distinct from **workflow** documents under `wiki/workflows/` (living integration maps: cron summaries, Trello/calendar wiring, helper registry, runtime layout).
- **Marcos rulebook**: Marcos’s durable maintenance instructions as three **vault documents** under `wiki/runbooks/marcos/`: `master.md`, `scope-rules.md`, `nightly-prompt.md`. The `marcos/` **agent workspace** holds only OpenClaw machinery (persona, `memory/`, `skills/`, a short `AGENTS.md` that points at the rulebook and report paths)—not duplicate copies of the rulebook or dated audit outputs.
- **vault document**: any **produced** artifact (reports, runbooks, job materials, project writeups, screenshots with context, research, admin notes, etc.) that is not required as OpenClaw workspace machinery for a specific agent. Before curation it appears under **raw input**; after curation it lives under **wiki/** for the Obsidian document corpus. Distinct from per-agent **agent workspace** files OpenClaw needs in that agent’s tree: persona roots (`AGENTS.md`, `SOUL.md`, …), `memory/`, `skills/`, `hooks/`, `taskflow/`, agent-specific CSVs, and similar.
- **produced artifact**: output or durable reference material created by an agent or operator during work (nightly report, application draft, audit, export note)—as opposed to configuration or state OpenClaw loads by convention from the agent workspace.
- **wiki layout depth**: paths under `wiki/` may be at most **`wiki/{bucket}/{file}.md`** or **`wiki/{bucket}/{subfolder}/{file}.md`** — one optional subfolder under a **wiki top-level folder**, never deeper. Cheryl must not create `wiki/bucket/a/b/file.md`. Existing human-facing corpus already conforms (~124 pages flat in bucket, ~40 with one subfolder).
- **wiki top-level folder**: a synced directory bucket directly under `wiki/` (lowercase, no spaces). Core buckets: `reports/`, `runbooks/`, `job-search/`, `personal/`, `projects/`, `workflows/`, `contradictions/` (temporary open conflicts). Cheryl may **create new top-level folders** when **wiki ingest** needs a home for entity, concept, or synthesis pages (e.g. `entities/`, `concepts/`) — prefer **combining** into an existing page or bucket when possible before adding a bucket. `openclaw-docs/` is excluded from curation and layout rules (agent mirror only).
- **dated report folder**: under `wiki/reports/`, a subdirectory named `YYYY-MM-DD` holding agent-generated dailies (e.g. nightly job search). One-off research and audits live at `wiki/reports/` root or in topic subfolders (e.g. `repo-maintenance/`), not under date folders unless tied to a specific day.
- **legacy export capture**: screenshot/PNG files from the old `Docs/Exports and Backups/` tree are not migrated into the vault; they remain only in the `workspace.old` backup (or are discarded). New captures are embedded in markdown **vault documents**, not stored as standalone export files.
- **job-search document**: a **vault document** for applying—résumés, cover letters, application drafts, follow-ups, application status. Lives under `wiki/job-search/`. Distinct from **job-search report** (nightly analysis, shortlists, market research, job-related RCAs under `wiki/reports/` or `wiki/reports/YYYY-MM-DD/`).
- **wiki bootstrap**: one-time import of legacy `Docs/` and stray `wiki/` root files into the agreed `wiki/` layout by deterministic copy (not via **raw input**), so the **wiki curator** is not overloaded on day one. After bootstrap, new **produced artifacts** from agents use **raw input** only.
- **wiki migration dedup**: during **wiki bootstrap** (and when the curator files a drop), each file lands in its target **wiki top-level folder**; if a copy already exists elsewhere in `wiki/`, diff/merge content, then remove the duplicate—do not preserve pre-layout paths for convenience.
- **session notes**: episodic markdown under an agent workspace's `memory/` directory (typically `memory/YYYY-MM-DD.md`) — session log, compaction flush output, and short-lived context. Distinct from **agent quick-reference** and **vault documents**.
- **agent quick-reference**: the root `MEMORY.md` in an agent workspace — a small, curated set of facts the agent should recall without searching the **wiki**. Each agent owns its own; Cheryl does not edit others' files during **wiki maintenance**.
- **memory audit**: an agent's periodic review of its own **session notes** and **agent quick-reference** — trim stale content, consolidate duplicates, and decide **memory promotions**. Does not edit **wiki/** or run **wiki lint** (Cheryl's wiki maintainer domain).
- **memory promotion**: moving a durable, cross-agent fact out of agent memory into **raw input** so Cheryl can **wiki ingest** it. Producers do not write **vault documents** under `wiki/` directly.
- **session memory rules**: standing instructions for how an agent handles memory during any session (not on a schedule). **Core trio (v1):** run `/compact` before designing a new workflow; persist durable facts to **agent quick-reference** and read back; **memory promote** cross-agent facts via **interaction capture**. Lives in `HEARTBEAT.md`. Does not include standing `memory_search` before every repetitive task (weekly **memory audit** covers drift).
- **session memory search**: OpenClaw config flag `memorySearch.experimental.sessionMemory` — lets agents search prior chat sessions, not only **agent quick-reference** and **session notes**. Enabled agent-wide as part of the memory hygiene work; pairs with compaction **memory flush** and **memory audit**.
- **memory audit skill**: per-agent OpenClaw skill that defines **memory audit job** procedure — one each for Ubi, Cheryl, and Marcos (`ubi-memory-audit`, `cheryl-memory-audit`, `marcos-memory-audit`). Shared **memory retention (rule-based)** text may live in each skill's `references/`; agent-specific boundaries (e.g. Cheryl must not run **wiki lint**) stay in that agent's `SKILL.md`.
- **memory audit job**: isolated cron run where an agent executes its **memory audit skill** on its own workspace. Six jobs total (weekly + monthly × three agents); see **memory audit schedule**.
- **weekly memory audit**: the lighter **memory audit job** cadence (Mondays) — scan the past week's **session notes**, trim stale lines from **agent quick-reference**, and **memory promote** cross-agent facts.
- **monthly memory audit**: the deeper **memory audit job** cadence (1st of month) — archive or promote aged content in **agent quick-reference**, review **session notes** older than thirty days, and remove one-off notes that never promoted.
- **memory audit notification**: delivery rules for **memory audit job** runs — weekly runs reply `NO_REPLY` unless **drift** is found; monthly runs always send a brief Telegram summary. **Drift** means stale facts removed, **memory promotions** created, or ambiguity needing the operator's call.
- **memory retention (rule-based)**: what **memory audit** keeps vs moves vs removes. **Stays in agent quick-reference:** active preferences and rules the agent applies every session (e.g. writing style, live Trello behavior), plus anything still referenced in the last thirty days. **Memory promote:** cross-agent facts, completed project outcomes, infra or runbook-shaped content — via **raw input**, not direct **wiki/** edits. **Remove or archive:** calendar dates in the past, one-off session debugging, superseded rules, completed migrations. **Unsure** → treat as **drift** (weekly Telegram). Long rule sets may promote to **wiki** with a one-line pointer left in **agent quick-reference**.
- **memory audit schedule**: six isolated cron jobs (Ubi, Cheryl, Marcos × weekly + monthly), twenty-minute timeout, **failure alerting** on all (Telegram after two consecutive failures). **America/Denver** start times: Ubi **1:00**, Marcos **3:15**, Cheryl **5:15** (weekly Mondays; monthly 1st). Weekly jobs no-op on the 1st (**monthly memory audit** replaces **weekly memory audit** that day). Staggered to clear Marcos nightly (2:30), nightly job search (4:09), and morning attention sweep (6:30). **Wiki curator** runs six times daily at **midnight, 9 AM, noon, 3 PM, 6 PM, 9 PM** MT (replacing every-fifteen-minutes).
- **Trello production pipeline**: the production code path that receives Trello events, classifies them, queues unattended work, runs Trello/calendar automation, and applies resulting Trello/calendar changes.
- **Scheduled Trello/calendar job**: an unattended Trello/calendar automation job triggered by time or cron rather than by an incoming webhook event.
- **Pipeline state**: durable operational state used by the Trello production pipeline, such as queue records, handled-action tracking, retry tracking, and similar service state. This is distinct from agent memory or taskflow state.
- **Documentation contract test**: a `node:test` case in `UbiClawBot` that asserts a structural or cross-repo documentation invariant (for example, a forbidden duplicate path is absent or a required canonical page exists). These tests may read sibling agent repositories; they are not tests of runtime Trello behavior.

## Vault-native task system

- **task file**: a markdown file under `tasks/` at the vault root — frontmatter carries `status` (maps 1:1 to former Trello list names), `due`, `agent`, and other structured fields. Source of truth for the task state machine in the vault-native task system. Replaces Trello cards.
- **task vault**: the `tasks/` top-level folder at vault root — sibling to `wiki/` and `raw-input/`. Holds live task state only; archived tasks leave via **raw input** → Cheryl → **wiki**. Not a **wiki top-level folder**.
- **task gateway**: the thin local HTTP service replacing `trello-gateway` — same HTTP interface (`agentId`, `operation`, `cardId`, `params`), same transition matrix and per-agent authorization rules, backed by **task files** instead of the Trello API. Agents require zero changes to call it.
- **task watcher**: filesystem daemon monitoring `tasks/` for frontmatter `status:` changes, dispatching the existing pipeline automation (reschedule, missed, done calendar adjustments). Replaces Trello webhooks as the event trigger.

## Trello board language

- **open-card contract**: the gateway-enforced shape of cards on scoped lists—four description sections in fixed order (`Original Request`, `Research`, `Peer Review`, `Work completed`) plus section mutation rules (e.g. Marcos-only `Peer Review`). Distinct from agent prompts and skills.
- **backlog intake**: the procedure Ubi runs when a new or moved card enters **Backlog**—style the card, fill the open-card contract sections, then execute, block, or hand off. Not applied to **PR review cards**.
- **peer review**: optional feedback from Marcos on a backlog card Ubi judges technical (code, infra, deploy, security, architecture). Ubi @mentions Marcos and leaves the card in **Backlog** until he replies; not a **Blocked** state.
- **PR review card**: a Trello card created by `github-pr-bridge` for an open pull request (title pattern like `Review PR #N`). Lands in **Backlog**; owned by Marcos for the GitHub review; Ubi backlog intake is skipped.
- **Next steps checklist** *(retired)*: a native Trello checklist formerly required by the open-card contract. No longer enforced; open cards may still carry legacy copies pending a one-time cleanup sweep (skip **Done** and Trello-closed cards).
